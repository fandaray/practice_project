# Boiler SCADA — учебный проект OPC UA

Симуляция бойлера смешения (горячая + холодная вода → выход) с публикацией
параметров через **OPC UA сервер** и панелью визуализации в духе промышленной
SCADA/HMI-системы: анимированная мнемосхема с уровнем воды, потоками в трубах
и положением клапанов.

## Состав проекта

```
opcua_boiler_scada/
├── boiler_core/
│   └── model.py          # физика: баланс массы/энергии, уровень/темп./давление
├── server/
│   └── opcua_server.py   # OPC UA сервер, крутит модель раз в секунду
├── logger/
│   └── history_logger.py # OPC UA клиент → пишет тренд в SQLite раз в 5 сек
├── scada_dashboard/
│   ├── app.py            # Flask backend (OPC UA клиент + REST API)
│   ├── templates/index.html
│   └── static/{css,js}   # анимированная мнемосхема (SVG) + логика опроса
├── data/                 # сюда пишется boiler_history.sqlite3
└── requirements.txt
```

## Технологические блокировки (interlocks)

Поверх ручных уставок оператора работают две автоматические защиты — они
считаются в `BoilerModel.step()` и принудительно переопределяют команду,
если предыдущее значение параметра вышло за порог. Есть гистерезис (порог
срабатывания выше порога сброса), чтобы блокировка не "дребезжала" на границе:

| Блокировка         | Срабатывает при      | Сбрасывается при     | Действие                                  |
|---------------------|-----------------------|------------------------|--------------------------------------------|
| `Interlocks/Overflow` | уровень ≥ 96 %       | уровень ≤ 90 %        | клапаны HotValve и ColdValve принудительно закрываются |
| `Interlocks/Overtemp` | температура ≥ 95 °C  | температура ≤ 88 °C   | нагреватель Heater принудительно выключается |

Фактическое (после блокировки) состояние публикуется отдельно в
`Boiler/Effective/{HotValve,ColdValve,Heater}` — оно может отличаться от
`Boiler/Setpoints/...`, если сработала защита. На панели это видно как значок
"A" рядом с клапаном/нагревателем плюс отдельный индикатор в блоке
**AUTO INTERLOCKS**; сам слайдер при этом продолжает показывать то, что задал
оператор, а не то, что физически происходит с клапаном.

## Автоматическое регулирование (LIC-101 / TIC-101)

Кроме ручного управления клапанами и нагревателем, есть два простых
пропорциональных регулятора - как термостат и регулятор уровня:

- **LevelAuto** - при включении регулятор сам держит уровень около
  `LevelSetpoint`: если уровень ниже цели, открывает горячий и холодный клапан
  поровну; если выше - открывает слив. Пока регулятор активен, ручные значения
  `HotValve`/`ColdValve`/`OutletValve` игнорируются - регулятор переписывает
  их сам.
- **TemperatureAuto** - при включении регулятор держит температуру около
  `TemperatureSetpoint`, управляя мощностью нагревателя (`Heater`). Верхний
  предел уставки ограничен порогом блокировки перегрева (95°C) - выставить
  недостижимую цель нельзя.

Оба регулятора - простые П-регуляторы с зоной нечувствительности (не пытаются
попасть в уставку==вплоть до долей процента, останавливаются в небольшом
коридоре около неё, чтобы не "дребезжать"). Блокировки переполнения/перегрева
работают поверх них точно так же, как поверх ручного управления - если
регулятор уровня пытается долить бак выше 96%, приток всё равно принудительно
закроется.

На панели это два переключателя (LIC-101, TIC-101) с уставками - при
включении соответствующие ручные слайдеры визуально гаснут (ими можно не
пользоваться, регулятор их всё равно перепишет).

## Установка

```bash
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

## Запуск (три отдельных процесса/терминала)

```bash
# 1. OPC UA сервер с моделью бойлера
python -m server.opcua_server

# 2. Логгер истории в SQLite (опционально)
python -m logger.history_logger

# 3. Веб-панель SCADA
python -m scada_dashboard.app
```

Панель откроется на **http://localhost:5050** — сползающие бегунки задают
открытие клапанов и мощность нагревателя, справа читаются уровень,
температура, давление и статус аварий. На мнемосхеме видно, как вода
заполняет ёмкость, меняет цвет по температуре, а трубы «текут» пунктиром
пропорционально открытию клапанов.

## Как это работает

1. `opcua_server.py` раз в секунду читает уставки клапанов/нагревателя из
   адресного пространства, прогоняет шаг модели `BoilerModel.step()` и
   публикует новые измерения и алармы.
2. `scada_dashboard/app.py` — сам по себе OPC UA **клиент**: читает те же
   теги по HTTP-запросу браузера (`/api/state`) и пишет уставки обратно
   (`/api/setpoint`), когда пользователь двигает бегунок.
3. `history_logger.py` — независимый OPC UA клиент, который просто пишет
   срез параметров в SQLite каждые 5 секунд для последующего анализа трендов.
